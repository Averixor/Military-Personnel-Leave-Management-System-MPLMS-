# Scheduler Engine

## Основная задача

Scheduler Engine ищет допустимые и стабильные варианты отпусков.

Он не обязан находить идеальный график. Его задача — вернуть лучшие допустимые варианты в пределах заданного времени.

## Главная функция

```python
find_acceptable_slots(
    person,
    desired_date,
    max_slots=10,
    max_shift_days=14,
)
```

## Базовый алгоритм

1. Построить окно дат вокруг desired_date.
2. Сформировать кандидатов.
3. Исключить кандидатов, конфликтующих с frozen leaves.
4. Проверить self-overlap.
5. Проверить annual leave rules.
6. Проверить interval >= 4 months между основными отпусками.
7. Проверить overlap limits.
8. Проверить hierarchy restrictions.
9. Проверить правило +/-2 дня для затронутых отпусков.
10. Посчитать conflict_score.
11. Отсортировать варианты.
12. Вернуть 3-10 лучших вариантов.

## Основные ограничения

### Первый основной отпуск

Первый отпуск в году должен содержать минимум 15 непрерывных календарных дней.

### Второй основной отпуск

Остаток дней может быть:

- одним блоком;
- двумя блоками.

Примеры:

- 15 + 15;
- 18 + 12;
- 15 + 7 + 8;
- 20 + 10.

### Интервал

Между концом одного основного отпуска и началом следующего должно быть не менее 4 месяцев.

### Нет накопления

Если второй отпуск не помещается в календарный год:

- он сгорает;
- не переносится;
- не накапливается.

### Будущие отпуска

Система хранит минимум 4 будущих основных отпуска на человека.

## Freeze Logic

Обычный отпуск замораживается за 20 дней до начала.

Для critical personnel freeze period увеличивается внутренней политикой.

Frozen leave нельзя:

- двигать автоматически;
- двигать вручную;
- изменять;
- пересчитывать.

Исключение: hard override super admin.

## Правило +/-2 дня

Каждый отпуск хранит `initial_starts_on`.

Другие люди не могут сдвинуть этот отпуск более чем на +/-2 дня от initial date.

Сдвиг не накапливается. Всегда сравнивается с исходной датой.

## Overlap Limits

Ступенчатая модель:

- normal_limit + 1 — максимум 12 дней;
- normal_limit + 2 — максимум 8 дней;
- normal_limit + 3 — максимум 4 дня;
- normal_limit + 4 — запрещено.

## Conflict Score

Единая метрика конфликтности.

Учитывает:

- количество затронутых людей;
- суммарный размер сдвигов;
- criticality_level затронутых людей;
- severity overlap;
- необходимость override;
- ручные вмешательства;
- близость к freeze period.

Примерная форма:

```text
score =
  affected_people_weight
+ shift_distance_weight
+ criticality_weight
+ overlap_severity_weight
+ override_weight
+ manual_review_weight
+ freeze_proximity_weight
```

Чем меньше score, тем лучше вариант.

## Nightly Replan

Фоновый процесс для:

- ребалансировки графика;
- генерации будущих отпусков;
- исправления конфликтов;
- подготовки admin review items.

Требования:

- snapshot перед запуском;
- ограничение времени выполнения;
- сохранение лучшего найденного результата;
- отсутствие auto-apply;
- audit trail;
- rollback on failure.
